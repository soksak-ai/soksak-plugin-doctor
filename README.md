# soksak-plugin-doctor

Integrity gate for soksak plugins. Run it before publishing — a non-zero exit blocks release.

It checks one plugin against the **core-published contract** (`contract.json`, generated by the
core from its own theme engine and permission list). The plugin ecosystem stays consistent because
every rule has a single source in the core, not copy-pasted per plugin.

## Usage

```
npx soksak-plugin-doctor [pluginDir]   # default: current directory
```

Wire it into a plugin so publishing is gated:

```json
{
  "scripts": { "prepublishOnly": "soksak-plugin-doctor", "doctor": "soksak-plugin-doctor" },
  "devDependencies": { "soksak-plugin-doctor": "github:soksak-ai/soksak-plugin-doctor" }
}
```

Build the plugin first (the check reads the bundled `main.js`), then `npm run doctor`.

## Rules

| Rule | What it enforces |
|------|------------------|
| `naming` | `id` matches `soksak-plugin-<segments>` (lowercase kebab) and equals the directory name |
| `permission` | every declared permission exists in the core contract (catches permissions the core removed, e.g. a token left over after a capability moved to a plugin) |
| `theme` | the bundle references only theme CSS variables the core actually emits — a reference to a host-token name the core does not provide (e.g. `--text`, `--surface`, `--accent`, `--bg2`) is a ghost: it silently falls back to a hardcoded colour and the core theme is not applied. Library/private variables (`--radix-*`, `--color-blue-500`, `--gap`, self-defined `--x:`) are ignored. |

## Contract source of truth

`contract.json` is **generated by the core**, not authored here. The core owns the theme engine
(`COLOR_SLOTS`) and the permission list (`PERMISSIONS`); it emits `contract.json` and a pin test
fails if the file drifts from the live core. This package vendors a copy (refreshed from the core's
`src/plugins/contract.json`), the same cache model the plugin registry uses.
